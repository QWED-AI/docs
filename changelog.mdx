## v5.0.4 ‚Äî `verify_identity` fail-closed
**Released: April 19, 2026** ¬∑ [GitHub PR #153](https://github.com/QWED-AI/qwed-verification/pull/153)

> `verify_identity()` now fails closed when numerical sampling matches but no formal proof exists ‚Äî returning `BLOCKED` with `is_equivalent: false` instead of `UNKNOWN` with `is_equivalent: null`.

### Security

- **`verify_identity` fail-closed** ‚Äî When symbolic simplification is inconclusive and the numerical sampling fallback finds agreement at all fixed test points, the engine previously returned `UNKNOWN` with `is_equivalent: null` and `confidence: 0.0`. This ambiguous result could be misinterpreted as partial equivalence. The engine now returns `BLOCKED` with `is_equivalent: false`, `method: "numerical_sampling_rejected"`, and `confidence: 0.0`, making it explicit that sampling-only agreement is not a valid proof.

### Migration

No action required if you already treat `UNKNOWN` results as unverified. If your code branches on `status == "UNKNOWN"` for identity verification results, update it to check for `"BLOCKED"` instead:

```python
result = engine.verify_identity("expr_a", "expr_b")

# Before
if result["status"] == "UNKNOWN":
    handle_inconclusive()

# After
if result["status"] == "BLOCKED":
    handle_rejected()
```

## v5.0.3 ‚Äî `verify_logic_rule` fail-closed
**Released: April 18, 2026** ¬∑ [GitHub PR #152](https://github.com/QWED-AI/qwed-verification/pull/152)

> The legacy `VerificationEngine.verify_logic_rule()` method now raises `NotImplementedError` instead of silently returning `None`. Callers must migrate to `LogicVerifier`.

### Security

- **`verify_logic_rule` fail-closed** ‚Äî Previously, `VerificationEngine.verify_logic_rule(rule, context)` was a no-op that silently returned `None`, which could be misinterpreted as a passing verification. It now raises `NotImplementedError` with the message `verify_logic_rule is deprecated and fail-closed; use LogicVerifier instead`. This ensures that any code path relying on the legacy method fails explicitly rather than producing ambiguous results.

### Migration

Replace any usage of `VerificationEngine.verify_logic_rule()` with the dedicated `LogicVerifier` class:

```python
# Before (no longer works)
from qwed_new.core.verifier import VerificationEngine
engine = VerificationEngine()
result = engine.verify_logic_rule(rule="if x > 5 then allow", context={"x": 10})

# After
from qwed_new.core.logic_verifier import LogicVerifier
verifier = LogicVerifier()
result = verifier.verify_conclusion(
    premises=["x is 10", "x > 5 implies allow"],
    conclusion="allow"
)
```

## v5.0.2 ‚Äî Default-deny for unknown tool approvals
**Released: April 18, 2026** ¬∑ [GitHub PR #150](https://github.com/QWED-AI/qwed-verification/pull/150)

> The tool approval system now blocks all unknown tools by default, regardless of their heuristic risk score. Previously, unknown tools with a low risk score were auto-approved.

### Security

- **Default-deny for unknown tools** ‚Äî Tool calls that are not on the safe operations list or the dangerous operations list are now blocked unconditionally. Before this change, unknown tools with a risk score below 0.3 were auto-approved. Now they require explicit allowlisting. The blocked response includes the tool name and its computed risk score for debugging.

### Migration

If your agents use custom tools that were previously auto-approved based on risk score, you must add them to the safe operations allowlist. Any tool not explicitly listed is now denied with a message like `Unknown tool 'your_tool' requires explicit allowlisting (risk_score=0.0)`.

## v5.0.1 ‚Äî Legacy CodeExecutor hard-blocked
**Released: April 18, 2026** ¬∑ [GitHub PR #149](https://github.com/QWED-AI/qwed-verification/pull/149)

> Permanently disables the legacy `CodeExecutor` class that used raw `exec()`. Any direct usage now raises a `RuntimeError` directing callers to `SecureCodeExecutor`.

### Security

- **Legacy CodeExecutor hard-blocked** ‚Äî `CodeExecutor.execute()` now raises a `RuntimeError` on every call, closing the last remaining path to in-process code execution. The public API endpoints (`/verify/stats`, `/verify/consensus`) are unaffected because they already use `SecureCodeExecutor` since v4.0.4. This change impacts self-hosted deployments that imported `CodeExecutor` directly.

### Migration

If you import `CodeExecutor` in custom code, replace it with `SecureCodeExecutor`. See the [security hardening guide](/advanced/security-hardening#legacy-codeexecutor-removed) for a migration example.

## v5.0.0 ‚Äî Enforcement boundary hardening
**Released: April 4, 2026** ¬∑ [GitHub Release](https://github.com/QWED-AI/qwed-verification/releases/tag/v5.0.0)

> Major release making QWED's verification boundary fail-closed, deterministic about what it proves, and harder to bypass under adversarial conditions. Consolidates 98 commits and 20 merged PRs since v4.0.1, including the full enforcement hardening series.

### Breaking changes

- **`INCONCLUSIVE` is a new verification status** ‚Äî Natural-language math responses now return `INCONCLUSIVE` when verifying LLM-translated expressions, because the translation step is non-deterministic. Downstream consumers must handle this status alongside `VERIFIED`, `ERROR`, and `BLOCKED`.
- **`BLOCKED` and `UNKNOWN` are explicit outcomes** ‚Äî These are no longer treated as generic failures. Consumers should map them to distinct UI states or retry logic.
- **`ActionContext` is mandatory for agent verification** ‚Äî All `verify_action` requests must include `conversation_id` and `step_number`. Requests without them are rejected with `QWED-AGENT-CTX-001`.
- **`security_checks` field removed** ‚Äî Exfiltration and MCP poison checks are now server-enforced and unconditional. The `security_checks` request field and the TypeScript SDK `checkExfiltration`/`checkMcpPoison` options no longer exist.
- **`/metrics` endpoints require admin authentication** ‚Äî `GET /metrics` and `GET /metrics/prometheus` now require an authenticated admin or owner role. Update monitoring integrations accordingly.
- **Docker required for stats and consensus verification** ‚Äî The in-process execution fallbacks have been removed. If Docker is unavailable, `/verify/stats` and `/verify/consensus` return HTTP 503.

### Security

- **Fail-closed verification boundary** ‚Äî Disabled unsafe in-process execution fallbacks; stats and consensus paths now require the secure Docker sandbox.
- **Logic verifier eval() removed** ‚Äî The raw `eval()` fallback in logic constraint parsing has been removed. If `SafeEvaluator` is unavailable, the verifier raises a `RuntimeError`.
- **Consensus rate limiting** ‚Äî `POST /verify/consensus` now enforces per-tenant rate limiting, matching other verification endpoints.
- **Consensus fact self-attestation removed** ‚Äî The fact engine no longer participates in automatic consensus engine selection, preventing self-referential verification loops.
- **Redis fail-closed** ‚Äî The sliding window rate limiter now denies requests on Redis errors instead of allowing them.
- **Timing-safe token verification** ‚Äî Agent token comparison uses `hmac.compare_digest` to prevent timing side-channel attacks.
- **Environment integrity enforcement** ‚Äî The API server runs `verify_environment_integrity()` at startup before database initialization.

### Trust boundary metadata

API responses now include a `trust_boundary` object that describes exactly what was verified and what was not. This gives you machine-readable transparency into the verification scope.

```json
{
  "trust_boundary": {
    "query_interpretation_source": "llm_translation",
    "query_semantics_verified": false,
    "verification_scope": "translated_expression_only",
    "deterministic_expression_evaluation": true,
    "formal_proof": false,
    "overall_status": "INCONCLUSIVE"
  }
}
```

See the [`POST /verify/natural_language`](/api/endpoints#post-verifynatural_language) endpoint reference for full field descriptions.

### Agent hardening

- **Action context mandatory** ‚Äî All `verify_action` requests must include `conversation_id` and `step_number`. Requests without them are rejected with `QWED-AGENT-CTX-001`.
- **Replay detection** ‚Äî Reusing a `(conversation_id, step_number)` pair is blocked (`QWED-AGENT-LOOP-002`).
- **Loop detection** ‚Äî Repeating the same action 3+ consecutive times triggers a denial (`QWED-AGENT-LOOP-003`).
- **In-flight step reservations** ‚Äî Prevents race conditions when multiple agent calls run concurrently.
- **Budget denial isolation** ‚Äî Budget-exceeded denials do not consume conversation state, so the agent can retry after the budget resets.
- **Stats exception masking** ‚Äî The Stats Engine no longer leaks internal exception details to API callers.

### SDK version bumps

- `qwed` (PyPI): `4.0.1` ‚Üí `5.0.0`
- `qwed_sdk` (Python): `2.1.0-dev` ‚Üí `5.0.0`
- `@qwed-ai/sdk` (npm): `4.0.1` ‚Üí `5.0.0`
- TypeScript SDK: Removed `security_checks` from agent verification helpers; `tool_schema` remains.

### Upgrade

```bash
pip install qwed==5.0.0
docker pull qwedai/qwed-verification:5.0.0
npm install @qwed-ai/sdk@5.0.0
```

<Warning>
If you use `POST /verify/natural_language` and check the top-level `status` field, you must now handle `INCONCLUSIVE` as a valid outcome. Use the `trust_boundary` object to inspect the detailed verification breakdown. For fully deterministic results, use `POST /verify/math` directly.
</Warning>

<Warning>
If you use agent verification, all requests must now include `conversation_id` and `step_number` in the context. See the [agent verification guide](/advanced/agent-verification) for details.
</Warning>

## v4.0.5 ‚Äî Stats exception masking
**Released: April 3, 2026** ¬∑ [GitHub PR #118](https://github.com/QWED-AI/qwed-verification/pull/118)

> Prevents the Stats Engine from leaking internal exception details to API callers. Clients now receive a generic error message when stats code generation fails.

### Security

- **Stats translation error masking** ‚Äî When stats code generation fails, the API now returns `"Internal verification error"` instead of the raw exception message. This prevents accidental exposure of file paths, API keys, or other sensitive data that may appear in exception text.
- **Server-side logging preserved** ‚Äî The exception type is still logged server-side for operator debugging, but the log no longer includes the full exception message.

## v4.0.4 ‚Äî Fail-closed execution enforcement
**Released: April 3, 2026** ¬∑ [GitHub PR #117](https://github.com/QWED-AI/qwed-verification/pull/117)

> Removes unsafe in-process execution fallbacks from statistical and consensus verification. All model-generated Python now runs exclusively in the secure Docker sandbox.

### Breaking changes

- **Docker required for stats verification** ‚Äî The Wasm and restricted Python fallback execution paths have been removed from the Stats Engine. If Docker is unavailable, `/verify/stats` returns HTTP 503 instead of executing code in-process.
- **Consensus Python engine uses Docker** ‚Äî The Python verification engine within consensus verification now runs through `SecureCodeExecutor` instead of the in-process `CodeExecutor`. If Docker is unavailable during `high` or `maximum` mode, `/verify/consensus` returns HTTP 503.

### Security

- **Fail-closed sandbox gating** ‚Äî Both the Stats Engine and the Consensus Engine refuse to execute model-generated code when the Docker sandbox is not available, preventing any downgrade to in-process execution.
- **Live Docker health checks** ‚Äî `SecureCodeExecutor.is_available()` now pings the Docker daemon on each request instead of relying on cached startup state, catching mid-operation Docker failures.

### API changes

- `POST /verify/stats` now returns HTTP 503 when the secure runtime is unavailable, and HTTP 403 when generated code is blocked by security policy.
- `POST /verify/consensus` now returns HTTP 503 when secure execution is required but Docker is unavailable.

- **Consensus fact self-attestation removed** ‚Äî The Fact engine is no longer automatically selected during consensus verification. It produced self-referential results without external context.

## v4.0.2 ‚Äî Security Cleanup
**Released: April 1, 2026** ¬∑ [GitHub PR #114](https://github.com/QWED-AI/qwed-verification/pull/114)

> Hardens expression evaluation, improves error handling across the SDK and consensus engine, and removes unused code flagged by CodeQL.

### Security

- **eval() fully eliminated** ‚Äî SymPy and Z3 expression evaluation now uses a custom AST-walking interpreter instead of `compile()` + `eval()`. The evaluator validates every AST node against a strict allow-list, then interprets the tree directly in a restricted namespace.
- **CodeGuard integration** ‚Äî Expressions are screened by `CodeGuard` (when available) as a second defense layer between AST validation and evaluation.
- **Keyword unpacking blocked** ‚Äî Call nodes with `**kwargs`-style keyword unpacking are now rejected during AST validation.

### Improvements

- **Exact SymPy arithmetic** ‚Äî Numeric literals are coerced to `sympy.Integer` / `sympy.Float` during evaluation, preventing floating-point drift in intermediate math comparisons.
- **Consensus failure recording** ‚Äî Unexpected errors during async engine aggregation are now logged and recorded as an `EngineResult` entry instead of being silently dropped.
- **Telemetry initialization** ‚Äî Replaced the `_initialized` flag with `@lru_cache` for one-time cached initialization.
- **SDK import cycle broken** ‚Äî `qwed_sdk.cache` no longer imports from `qwed_sdk.qwed_local` for terminal colors; it uses `colorama` directly with a plain-text fallback.
- **Integration imports hardened** ‚Äî Optional framework imports (LangChain, CrewAI, LlamaIndex) now default to `None` explicitly instead of using bare `except: pass`.

### Chores

- Removed unused locals and globals across SDK, core modules, examples, and scripts.
- Replaced module-level `print()` calls in `database.py` with `logger.debug()`.
- Added `ast.Tuple` and `ast.List` to the safe SymPy node type allow-list.
