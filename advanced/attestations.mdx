## What are attestations?

An **attestation** is a cryptographically signed proof that a verification occurred. It:

- Uses **ES256 (ECDSA P-256)** signatures
- Is formatted as a **JWT**
- Can be verified independently
- Can be stored on-chain

## Requesting attestations

```python
result = client.verify(
    "2+2=4",
    include_attestation=True
)

print(result.attestation)
# eyJhbGciOiJFUzI1NiIsInR5cCI6InF3ZWQtYXR0ZXN...
```

## Fail-closed contract

`create_verification_attestation()` never returns `None`. It always returns an
`AttestationResult` whose `status` is one of three explicit states. Callers
**must** check `result.is_issued` before treating the attestation as valid —
silent downgrades (treating a missing proof as success) are not possible.

### `AttestationResult`

| Field | Type | Description |
| --- | --- | --- |
| `status` | `AttestationStatus` | `ISSUED`, `BLOCKED`, or `UNVERIFIABLE`. |
| `token` | `str \| None` | JWT string. Present only when `status == ISSUED`. |
| `error_code` | `str \| None` | Machine-readable failure reason. |
| `error` | `str \| None` | Human-readable error detail (safe to log). |
| `is_issued` | `bool` | `True` only when `status == ISSUED`. Use this as the gate. |

### Status values

| Status | Meaning | When it occurs |
| --- | --- | --- |
| `ISSUED` | Attestation signed and ready to use. | Normal success path. |
| `BLOCKED` | Crypto is available but signing failed. | Key error, JWT library error, or any signing exception. |
| `UNVERIFIABLE` | Cannot attempt signing. | `cryptography` / `PyJWT` not installed. |

### Error codes

| `error_code` | Paired status | Caller action |
| --- | --- | --- |
| `SIGNING_FAILURE` | `BLOCKED` | Hard-block the response; do not return verdict as `VERIFIED`. |
| `CRYPTO_UNAVAILABLE` | `UNVERIFIABLE` | Install `cryptography` and `PyJWT`, or run in a mode that does not require proofs. |

### Caller pattern

Treat anything other than `ISSUED` as a hard failure. Do **not** proceed as
`VERIFIED` without a valid attestation token:

```python
from qwed_sdk.attestation import create_verification_attestation

result = create_verification_attestation(
    status="VERIFIED",
    verified=True,
    engine="math",
    query="2+2",
)

if not result.is_issued:
    # Fail-closed: refuse to surface an unproven verdict.
    raise RuntimeError(
        f"Attestation unavailable [{result.error_code}]: {result.error}"
    )

use(result.token)
```

<Warning>
Never branch on `result.token is not None` alone. Always gate on `result.is_issued`
so that future failure states cannot silently bypass your check.
</Warning>

## Key lifecycle and continuity

Each `IssuerKeyPair` records when it was generated and which continuity policy
it follows. This makes key rotation and process-restart events auditable.

| Field | Description |
| --- | --- |
| `generated_at` | Unix timestamp when the key pair was created. |
| `key_continuity_policy` | `"ephemeral"` (default) or `"persistent"`. |

- **`ephemeral`** — Key material lives only in process memory. A new key is
  generated on every restart. Use this for short-lived workers or local dev.
- **`persistent`** — Key material is durably stored outside the process
  (for example, in an external KMS). Use this when verifiers across restarts
  or replicas must accept the same issuer key.

```python
from qwed_sdk.attestation import IssuerKeyPair

key_pair = IssuerKeyPair(
    issuer_did="did:qwed:node:production",
    key_id="signing-key-2026",
    key_continuity_policy="persistent",
)
```

Any other value for `key_continuity_policy` raises `ValueError` — typos cannot
silently degrade audit signal. Every key generation emits a structured log
line for downstream auditing:

```text
attestation.key_generated issuer=did:qwed:node:production key_id=signing-key-2026 policy=persistent generated_at=1747526400
```

Signing failures and crypto-unavailable paths emit matching audit lines:

```text
attestation.blocked       query_hash=<sha256-prefix> reason=SIGNING_FAILURE
attestation.unverifiable  query_hash=<sha256-prefix> reason=CRYPTO_UNAVAILABLE
```

## Attestation structure

### Header

```json
{
  "alg": "ES256",
  "typ": "qwed-attestation+jwt",
  "kid": "did:qwed:node:production#signing-key-2024"
}
```

### Payload

```json
{
  "iss": "did:qwed:node:production",
  "sub": "sha256:abc123...",
  "iat": 1703073600,
  "exp": 1734609600,
  "jti": "att_xyz789",
  "qwed": {
    "version": "1.0",
    "result": {
      "status": "VERIFIED",
      "verified": true,
      "engine": "math",
      "confidence": 1.0
    },
    "query_hash": "sha256:def456...",
    "proof_hash": "sha256:ghi789..."
  }
}
```

## Verifying attestations

### Using the API

```python
valid, claims, error = client.verify_attestation(jwt)
if valid:
    print(f"Verified by: {claims['iss']}")
```

### Using the SDK

```python
from qwed_sdk import verify_attestation

is_valid = verify_attestation(
    jwt="eyJhbGci...",
    trusted_issuers=["did:qwed:node:production"]
)
```

## Trust anchors

QWED maintains a registry of trusted attestation issuers:

| Issuer DID | Name | Status |
| --- | --- | --- |
| `did:qwed:node:production` | QWED Production | ✅ Active |
| `did:qwed:node:staging` | QWED Staging | ✅ Active |

## Attestation chains

Link multiple attestations together:

```python
attestation1 = client.verify("step1", include_attestation=True)
attestation2 = client.verify(
    "step2",
    include_attestation=True,
    chain_id="chain_abc",
    chain_index=1,
    previous_attestation=attestation1.jti
)
```
