The primary usage pattern for QWED-Tax is the **Pre-Flight Check**.

## TaxPreFlight Middleware

Wrap your Fintech API calls with this auditor.

```python
from qwed_tax.verifier import TaxPreFlight

# 1. Initialize
preflight = TaxPreFlight()

# 2. Capture Intent (from AI Agent)
intent = {
    "action": "hire_worker",
    "worker_type": "1099",  # LLM Decision
    "worker_facts": {
        "provides_tools": True,       # Fact: We gave them a laptop
        "reimburses_expenses": True,  # Fact: We pay for travel
        "indefinite_relationship": True
    },
    "state": "NY",
    "sales_data": {"amount": 600000}  # Fact: YTD Sales
}

# 3. Audit
report = preflight.audit_transaction(intent)

# 4. Enforce
if not report["allowed"]:
    print(f"🛑 BLOCKED: {report['blocks']}")
    # Do NOT call Gusto/Stripe API
else:
    print("✅ Verified. Proceeding to Fintech API.")
    # call_gusto_api()
```

### Capital gains date validation

The `audit_transaction` method now validates that both `buy` and `sell` dates are present when `asset_type` and `dates` are provided in the intent. If either is missing, the transaction is blocked with a descriptive error:

```python
report = preflight.audit_transaction({
    "asset_type": "equity",
    "dates": {}  # Missing buy and sell dates
})
# report["allowed"] = False
# report["blocks"] = ["Capital gains checks require both buy and sell dates."]
```

### Standalone usage

You can also use specific guards individually found in `qwed_tax.jurisdictions`.

```python
from qwed_tax.jurisdictions.us import PayrollGuard

pg = PayrollGuard()
result = pg.verify_fica_tax(gross_ytd=180000, current=5000, claimed_tax=310)
print(result.message) 
# -> "❌ FICA Error: Expected $68.20 (Hit Limit)"
```

## Gusto payroll middleware

The `QWEDTaxMiddleware` class intercepts AI-generated payroll payloads destined for execution engines like Gusto. It validates the payload schema and mathematically verifies gross-to-net tax logic before allowing the API call to proceed.

Use this when your AI agent generates payroll commands that need deterministic verification before execution.

```python
from qwed_tax.middleware.gusto_interceptor import QWEDTaxMiddleware

middleware = QWEDTaxMiddleware()

# AI-generated payload meant for Gusto API
ai_payload = {
    "payroll_entry": {
        "employee_id": "E001",
        "gross_pay": "5000.00",
        "taxes": [
            {"name": "Federal Income Tax", "amount": "750.00"},
            {"name": "Social Security", "amount": "310.00"}
        ],
        "deductions": [
            {"name": "401k", "amount": "250.00", "type": "PRE_TAX"}
        ],
        "net_pay_claimed": "3690.00",
        "currency": "USD"
    }
}

decision = middleware.process_ai_payroll_request(ai_payload)

if decision["execution_permitted"]:
    # Safe to forward to Gusto API
    gusto_api.execute(decision["validated_payload"])
else:
    # Block the request — AI hallucinated the math
    log_error(decision["reason"])
```

The middleware returns one of three statuses:

| Status | Meaning |
| --- | --- |
| `VERIFIED` | Tax logic is mathematically sound. Safe to execute. |
| `BLOCKED` (TAX_LOGIC_HALLUCINATION) | AI-generated math is incorrect. Request is blocked. |
| `BLOCKED` (INVALID_PAYLOAD) | Payload is missing required fields or fails schema validation. |

<Note>
  The middleware initializes a US jurisdiction `TaxVerifier` by default. It validates payroll entries using the `PayrollEntry` Pydantic model, so your payload must conform to the expected schema (see `qwed_tax.models`).
</Note>

## TypeScript SDK

Run compliance checks proactively in the browser/frontend.

Run compliance checks proactively in the browser/frontend.

```bash
npm install @qwed-ai/tax
```

```typescript
import { TaxPreFlight } from '@qwed-ai/tax';

const result = TaxPreFlight.audit({
  action: "hire",
  worker_type: "1099",
  worker_facts: { provides_tools: true, reimburses_expenses: true } // implies Employee
});

if (!result.allowed) {
   alert(" Compliance Block: " + result.blocks.join(", "));
}
```
